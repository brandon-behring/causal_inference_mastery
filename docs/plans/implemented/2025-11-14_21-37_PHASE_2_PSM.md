### Phase 2.2: Nearest Neighbor Matching (6-7 hours)

**Objective**: Implement NN matching with/without replacement, Abadie-Imbens SE

#### Tasks:
- [ ] **2.2.1**: Implement NN matching WITHOUT replacement
  - Greedy 1:1 matching on propensity score
  - Distance metric: absolute difference in propensity
  - Return matched pairs

- [ ] **2.2.2**: Implement NN matching WITH replacement
  - Allow control units to match multiple treated units
  - Track number of times each control matched (for Abadie-Imbens variance)

- [ ] **2.2.3**: Implement Abadie-Imbens (2008) analytic variance
  - **CRITICAL**: Only valid for matching WITH replacement
  - Formula accounts for matching uncertainty
  - Document why bootstrap FAILS for with-replacement matching

- [ ] **2.2.4**: Implement pairs bootstrap SE
  - **CRITICAL**: Only valid for matching WITHOUT replacement
  - Resample matched pairs
  - Document when each SE method is appropriate

- [ ] **2.2.5**: Known-answer tests
  - Perfect match (propensity identical) → ATE = difference in means
  - No common support → should error or warn

- [ ] **2.2.6**: Adversarial tests (10+)
  - No matches found (disjoint propensity distributions)
  - All units get matched to same control (with replacement)
  - Ties in propensity scores (multiple equally good matches)
  - Extreme propensity scores (near 0 or 1)

**Expected Output**: NN matching estimator with proper SE methods and 10+ adversarial tests

### Phase 2.3: Caliper & Kernel Matching (5-6 hours)

**Objective**: Implement caliper and kernel matching variants

#### Tasks:
- [ ] **2.3.1**: Implement Caliper Matching
  - Nearest neighbor within caliper distance (e.g., 0.1 × SD(propensity))
  - Units outside caliper dropped (affects sample)
  - Document trade-off: better balance vs sample size

- [ ] **2.3.2**: Implement Kernel Matching
  - Weighted average of control outcomes
  - Epanechnikov or Gaussian kernel
  - Bandwidth selection (rule of thumb or cross-validation)

- [ ] **2.3.3**: Known-answer tests
  - Large caliper (includes all) → equivalent to NN matching
  - Kernel with infinite bandwidth → simple mean difference

- [ ] **2.3.4**: Adversarial tests (10+ each)
  - Caliper: No units within caliper (all dropped)
  - Kernel: Bandwidth = 0 (only exact matches)
  - Kernel: Bandwidth → ∞ (all units weighted equally)

**Expected Output**: 2 additional matching estimators with tests

### Phase 2.4: Balance Diagnostics (4-5 hours)

**Objective**: Implement SMD, variance ratios, Love plots for balance assessment

#### Tasks:
- [ ] **2.4.1**: Implement Standardized Mean Difference (SMD)
  - Formula: SMD = (mean_treated - mean_control) / pooled_SD
  - **CRITICAL**: Calculate for ALL covariates, not subset
  - Threshold: |SMD| < 0.1 (Austin & Stuart 2015)

- [ ] **2.4.2**: Implement Variance Ratio
  - Formula: VR = var_treated / var_control
  - Threshold: 0.5 < VR < 2.0
  - Check for each covariate

- [ ] **2.4.3**: Implement Love Plot
  - X-axis: Covariate names
  - Y-axis: Absolute SMD
  - Two series: Before matching, After matching
  - Horizontal line at SMD = 0.1 threshold

- [ ] **2.4.4**: Create balance table
  - Rows: Covariates
  - Columns: Mean_treated, Mean_control, SMD, VR, Pass/Fail
  - Summary: % covariates balanced

- [ ] **2.4.5**: Known-answer tests
  - Perfect balance → SMD = 0, VR = 1
  - No overlap → Large SMD

- [ ] **2.4.6**: Integration with matching estimators
  - Run balance diagnostics automatically after matching
  - Warn if balance fails (SMD > 0.1 for any covariate)

**Expected Output**: Balance diagnostic toolkit integrated with estimators

**Issue**: Bootstrap invalid for matching WITH replacement (Abadie-Imbens 2008)

**Mitigation**:

- Implement pairs bootstrap ONLY for without-replacement matching
- Document WHEN each SE method is valid
- Add tests showing bootstrap coverage failure for with-replacement

### MEDIUM-5: Balance Diagnostics Not Comprehensive ⚠️
**Issue**: Must verify balance on ALL covariates, not subset

**Mitigation**:
- Calculate SMD for every covariate in dataset
- Report max|SMD| across all covariates
- Use SMD < 0.1 threshold (Austin & Stuart 2015), not 0.05 (too strict)
- Create Love plots showing all covariates
- Fail if ANY covariate has |SMD| > 0.1

## Validation Requirements

From `PHASE_COMPLETION_STANDARDS.md` and `ROADMAP.md`:

### Six-Layer Validation Architecture (MANDATORY)

1. **Layer 1: Known-Answer Tests** (2+ per estimator, 8+ total)
   - Perfect balance scenarios
   - No overlap scenarios
   - Propensity model correct vs misspecified

2. **Layer 2: Adversarial Tests** (10+ per estimator, 40+ total)
   - No common support
   - Perfect separation
   - Extreme imbalance (99:1)
   - Missing data
   - High-dimensional (p > n)
   - Multicollinearity

3. **Layer 3: Monte Carlo Validation** (10,000 sims)
   - Bias < 0.05
   - Coverage 94-96% for α = 0.05
   - **NEW**: Multi-alpha coverage (α ∈ {0.01, 0.05, 0.10})
   - SE accuracy < 10%
   - Test BOTH Abadie-Imbens and bootstrap SEs

4. **Layer 4: Python-Julia Cross-Validation**
   - Estimate agreement < 0.001
   - SE agreement < 5%

5. **Layer 5: R Triangulation**
   - Independent R implementation
   - Agreement verification
   - Graceful degradation if R unavailable

6. **Layer 6: Golden Reference (LaLonde 1986)**
   - NSW training program data
   - Compare to published ATE estimate
   - Compare PSM to RCT benchmark

### Additional Requirements

- **Test Coverage**: >80% line coverage
- **Documentation**: 8-section docstrings for all estimators
- **Benchmarks**: Performance baselines for n ∈ {100, 500, 1000, 5000, 10000}
- **Error Handling**: Explicit error messages with solutions

## Risk Assessment & Mitigation

### High-Risk Areas

1. **Abadie-Imbens Variance Formula** (HIGH)
   - **Risk**: Complex formula, easy to implement incorrectly
   - **Mitigation**: Validate against R `MatchIt` package, cross-check with original paper

2. **Bootstrap Coverage for With-Replacement** (HIGH)
   - **Risk**: Bootstrap produces incorrect SEs (Abadie-Imbens 2008 result)
   - **Mitigation**: Monte Carlo test showing coverage failure, document prominently

3. **Common Support Violations** (MEDIUM)
   - **Risk**: Units outside overlap region, unclear how to handle
   - **Mitigation**: Explicit policy (drop units, warn user), document in Balance Diagnostics

4. **LaLonde Replication** (MEDIUM)
   - **Risk**: May not match published results exactly (different implementation details)
   - **Mitigation**: Document discrepancies, explain differences (caliper, SE method, etc.)
